# 전체판 사용법

*한국어 | [English](USAGE.en.md)*

> 이 문서는 **전체판**을 돌리고 그 결과를 읽는 법입니다. *짧은 글 빠른 점검*은 [라이트 사용법](USAGE_lite.md), *어떤 도구를 언제* 쓸지는 [README](README.md)에 있습니다.

**한 바퀴는 이렇습니다.** 프롬프트를 붙이고 → 분석할 글을 붙이면 → 도구가 논증을 펼쳐 *1차 분석*을 내고 **멈춥니다** → 내가 깊이 볼 곳을 고르면 → 그곳만 *2차*로 자세히 → **판단은 내가** 합니다. 설치는 없습니다.

---

## 1. 실행하기 — 넷이면 됩니다

**복사 두 번, 붙여넣기 한 번, 그 다음은 도구가 묻는 대로.**

1. **챗봇을 엽니다.** chatgpt.com이나 claude.ai에서 새 대화를 엽니다(무료로도 됩니다).
2. **프롬프트 두 파일을 붙입니다.** [`prompts/LARP.md`](prompts/LARP.md) 내용을 *통째로* 복사해 첫 메시지로 붙이고, [`prompts/LARP_modules.md`](prompts/LARP_modules.md)도 *이어서 함께* 붙입니다(길어도 괜찮습니다). 자주 쓰면 챗봇의 '프로젝트'나 '맞춤 지시'에 한 번 넣어 두면 매번 붙일 필요가 없습니다.
3. **분석할 글을 통째로 붙입니다.** 긴 판결문·논문도 그대로 — 도구가 먼저 **글이 다투는 문제(쟁점)의 목록을 보여주고 멈춥니다.** 궁금한 것을 말로 고르세요 — "F 진술을 믿은 게 맞나"처럼. 번호·기호는 몰라도 됩니다. (짧은 글이면 이 단계 없이 바로 분석이 나옵니다.)
   - 확인 하나: 붙여넣은 텍스트에 `- 12 -` 같은 **쪽 표시가 살아 있으면** "몇 쪽을 펴 보라"는 안내까지 받습니다.
   - 챗봇이 "글이 너무 길다"며 잘라먹으면 → **§4**의 차선책(쟁점 대목만 넣기).
4. **결과를 읽고, 더 팔 곳을 고릅니다.** 도구는 *쉬운 요약*부터 내고, 의심스러운 자리를 짚은 뒤 **멈춥니다**(읽는 법은 **§2**). "계속"이라 하면 핵심(★)만 알아서 깊이 보고, 아니면 말로 지정하세요(**§3**). 마지막에 "보고서로 정리해줘"라고 하면 *읽는 사람의 순서*로 다시 쓴 최종 문서를 줍니다 — ① 이 문서가 말하는 것 ② 결론이 실제로 기대는 곳 ③ 튼튼한 곳 ④ 위험한 곳 ⑤ 무엇이 확인되면 어떻게 되나 ⑥ 사람 몫의 판단. 판정은 도구가 아니라 당신이 합니다.

> 아주 긴 문서의 사전 정리(게이트0)와 챗봇 용량이 작은 환경(분할판)은 **§5**의 고급 기능으로 옮겨 두었습니다 — 평소엔 필요 없습니다.

**〈넣을 것〉 양식 (선택).** 글만 넣어도 되지만, 아래를 같이 적으면 더 정확합니다. 빈 칸은 도구가 "자료 없음"으로 표시하고 진행합니다.

```text
[분석 대상]   "○○ 제작자는 사기다"라는 온라인 고발 글
[검토할 주장] "제작자는 처음부터 만들 생각 없이 후원금을 가로챘다"
[가진 근거]   후원금 입금 내역, 미배송 사실, 돈을 어디에 썼는지 정황
[분석 목적]   주장 검토
```

## 2. 결과 읽는 법 — 1차에 무엇이 나오나

> **참고 — 도구는 '장면'이라는 단계로 움직입니다.** ①**장면 1(지도)**: 결론과 근거의 뼈대를 그려 주고 *멈춥니다*. → 내가 볼 범위를 고르면 → ②**장면 3(선별)**: 그 부분에서 미심쩍은 논증(도구 용어로 '이상 논증')만 짚습니다. → ③원하면 **장면 4(보고서)**로 읽기 좋은 순서의 최종 문서를 받습니다. "1차/2차"는 이 흐름을 쉽게 부른 이름입니다.

**맨 앞에 *쉬운 요약*(코드·전문용어 없는 평서문)이 먼저 나옵니다 — 그것부터 읽으세요.** "이 글의 결론은 무엇인가, 말 없이 깔고 있는 가정은 무엇인가, 결정적으로 보이는 근거가 정말 결론을 정해 주는가, 무엇이 빠져 있는가"를 쉬운 말로 정리해 줍니다. 아래는 그 뒤에 붙는 *상세*를 읽는 법입니다.

### (1) 논증 블록
논증(주장과 그 근거의 짜임)마다 도구가 정리해 주는 짧은 칸입니다. 먼저 볼 셋:

- **숨은 전제** — 글이 말 안 하고 몰래 기대는 가정(`암묵` 표시).
- **겉과 속이 다른 곳(갈림)** — 글이 내세운 이유와, 실제로 결론을 떠받치고 있는 이유가 다른 곳. 제일 먼저 볼 곳.
- **여섯 질문 결과** — "아니다 / 모르겠다"가 나온 번호. 의심의 이유가 거기 있습니다.

### (2) 논리 지도 (기호)

| 기호 | 뜻 |
|---|---|
| C / A / E | 결론 / 주장 / 증거 (글에 적힌 것) |
| W | 숨은 전제 (글엔 없음 · 도구가 채워 넣음) |
| L | 그렇게 보이게 만든 조건 (글엔 없음 · 도구가 채워 넣음) |
| H | 다른 설명 (글엔 없음 · 도구가 채워 넣음) |
| V | 있어야 하는데 없는 증거 |
| ★ | 결론에 가장 중요한 것 (안 고르면 이것만 2차로) |

### (3) 증거 목록 (도구는 '대장'이라 부릅니다)
인용·언급된 증거가 *하나도 안 빠지고* 한 줄씩 올라옵니다. 각 줄에 — *어디서 나온 증거인가*(본인이 한 말 / 남에게 전해 들은 말 / 문서 같은 객관 자료), *같은 출처끼리 묶음*(같은 사람 말은 한 번만 셈), *가르는 힘이 있나*(그 증거만으로 어느 설명이 맞는지 판가름되나 — 아래 (4)), *원본이 맞는지 다툼이 있나*. 도구가 "인용은 됐는데 안 다뤘다"고 짚은 *빠뜨린 후보*도 여기 표시됩니다 — 무엇이 비었는지가 한눈에 보입니다.

### (4) 증거-가설 표 (설명이 두 가지로 나뉠 때; 도구는 '행렬'이라 부릅니다)
"사기냐 실수냐"처럼 설명이 두 가지로 나뉘는 글이면, 증거(가로줄)와 설명들(세로칸)을 맞춰 보는 표가 나옵니다.

- 칸: `+` 그 설명에 맞음 / `−` 어긋남 / `0` 상관없음 / `±` 읽는 방법에 따라 다름.
- **양쪽에 다 맞는 증거** = 어느 설명에나 맞아서 *어느 쪽이 맞는지 판가름해 주지 못하는* 증거(예: "돈을 보냈다"는 사실은 사기여도 실수여도 맞음). 도구는 이걸 '비진단'이라 부릅니다 — 이런 증거를 핵심 근거로 쓰면 약합니다.
- **혼자서 한쪽 손을 들어 주는 증거** = 한 설명에만 맞고, 다른 증거와 출처가 겹치지도 않는 것. 결론이 이런 증거를 갖고 있는지가 가장 중요합니다.
- **비어 있는 증거(V)** = 그 설명이 맞다면 있어야 하는데 자료에 없는 것. 어느 설명에 구멍이 몰렸는지 보세요.

한마디로 — *그럴듯하게 들어맞는지*가 아니라 *어느 쪽이 맞는지 판가름해 주는지*로 읽습니다.

> 표 머리의 **빠짐 여부 표시**도 꼭 보세요 — *증거 전부 반영*인지 *빠짐 검사 전(임시)*인지. **증거가 빠진 채의 평가는 임시 결과**입니다(빠진 한 건이 판가름 증거라면 결론이 뒤집힐 수 있으니까요). 검사 전이면 §5.3으로 증거가 다 들어왔는지 먼저 확인하세요.

### (5) 글 전체에 대한 의견 (세 가지 신호)
끝에 글 전체에 대한 종합 의견이 나옵니다 — "세 가지 신호(말 안 한 가정이 한쪽에만 쏠려 있다 · 양쪽에 다 맞는 증거에 기대고 있다 · 어떤 반대 자료가 나와도 안 흔들리게 짜여 있다)가 모두 한 방향이면, 증거를 보기 전에 결론부터 정해 둔 글일 수 있다"는 식입니다.

→ 그다음 도구는 멈추고 "어디를 더 볼까요?"를 묻습니다(**§3**).

## 3. 깊이 볼 곳 지정 (2차)

**말로 하면 됩니다.** "피고인이 시켰다는 부분이 정말 근거가 있나", "F 진술을 믿은 게 맞나"처럼 궁금한 것을 그대로 적으세요 — 도구가 해당 논증을 찾아 "이렇게 이해했습니다 (○쪽). 맞습니까?"로 확인하고 진행합니다. 그냥 "계속"이라고 하면 핵심(★)만 알아서 봅니다.

지도의 기호가 익숙하면 그걸로 지정해도 됩니다:

```text
A1, W1, V1      ← 특정 항목 지정
E1->A1          ← 특정 연결 지정
논증 후보 2      ← 번호로 지정
```

특정 문장 하나만 빠르게 보고 싶으면: `이 논증만 봐줘. 지도는 건너뛰고. "...원문 문장..."`

2차에서도 맨 앞에 *쉬운 요약*(무엇이 달라졌고 무엇이 남았나)이 먼저 나옵니다.

## 4. 글이 너무 길어 잘릴 때 — 쟁점 대목만 넣기 (차선책)

**기본은 통째로 붙이는 것입니다(§1).** 도구가 쟁점 목록을 보여주고, 당신이 고른 쟁점만 깊이 파는 방식이 정확도도 가장 좋았습니다(쟁점 하나로 좁히면 작은 모델도 놓치는 것이 거의 없습니다).

다만 챗봇의 입력 한도가 작아 **긴 문서가 잘려 들어가는 환경**이라면, 사람이 나눠 주는 차선책을 씁니다:

1. **쟁점 하나의 대목만 오려 넣습니다.** 판결문이면 목차에서 그 죄목·쟁점의 항(예: "6. 스마트팜 비용 대납 …판단")이 시작하는 쪽부터 다음 항 직전까지. 어디부터인지 모르겠으면 목차 부분만 먼저 붙이고 "○번 쟁점이 몇 쪽부터인지 알려줘"라고 물어도 됩니다.
2. **(선택) 더 정밀한 지도가 필요하면.** 전체판만으로도 쟁점 지도가 먼저 나옵니다. 증거 번호표 대조·증거대장까지 갖춘 정밀 지도를 원할 때만 [LARP-Map 긴 문서 모드](prompts/LARP_map_long.md)를 따로 씁니다.
3. **그 대목을 §1대로 전체판에 돌립니다.** 증거 목록과, 설명이 두 가지로 나뉘면 증거-가설 표(§2)까지 나옵니다.
4. **빠진 곳은 결과에 이미 표시됩니다.** *비어 있는 증거(V)*·*빠뜨린 후보*를 도구가 스스로 짚어 줍니다.
5. **다음 쟁점으로, 마지막에 합칩니다.** 쟁점을 가로지르는 연결(한 자료가 두 쟁점에 겹쳐 쓰임 등)은 사람이 잇습니다.

**"한 번에 모든 증거·모든 가설"은 도구의 약속이 아닙니다** — 한 번에 다 펼치는 건 사람도 못 합니다. 빠짐없음은 *한 번의 실행*이 아니라 *쟁점을 하나씩 나눠 보는 절차*에서 나옵니다.

## 5. 더 정확히 — 선택 기능

평소엔 §1~§4로 충분합니다. 아래는 *더 확실히* 하고 싶을 때만. 특히 — **도구가 드러낸 근거마다 그대로 딥리서치에 넣을 질문을 만들어 주는 것이 이 도구의 핵심 쓰임새 중 하나입니다**(§5.1 검증용 · §5.2 획득용). 숨은 전제를 짚는 데서 그치지 않고 *무엇을 확인할지*까지 손에 쥐여 줍니다 — 답과 판정은 여전히 사람 몫이고요.

### 5.0 아주 긴 문서의 사전 정리, 챗봇 용량이 작은 환경 (고급)

**아주 긴 문서(판결문 등)의 사전 정리 — 게이트0.** 분석 *전에* [게이트0](prompts/LARP_gate0.md)로 기계적인 훑기를 먼저 하세요 — 워터마크·쪽번호·*지워진 인용부(리댁션)*·증거 꼬리표를 훑어 놓으면, 이후 분석이 놓친 것을 대조할 *기준표*가 생깁니다(코드 실행 환경이면 [`tools/larp_gate0.py`](tools/larp_gate0.py)가 더 정확).

**분할판 실행 순서 (컨텍스트가 작아 큰 프롬프트가 깨지는 환경 — NotebookLM 등).** *어느 파일을 어느 단계에 넣는지*가 핵심입니다.

1. **지도 단계:** [S0 공통규율](prompts/LARP_split_S0_common.md) + [S1 지도](prompts/LARP_split_S1_map.md)를 소스로 넣고, 먼저 [게이트0](prompts/LARP_gate0.md)로 문서를 훑어 정리합니다(코드 환경이면 [`tools/larp_gate0.py`](tools/larp_gate0.py), 아니면 게이트0의 수동 다섯 훑기 — 코드 전용이 아닙니다). → 문서를 붙이면 → *구조 지도*만 그리고 **멈춥니다.** 이 단계에선 *일부러 문제를 짚지 않습니다* — 선별 기준(증상 색인·6문)이 아직 로드되지 않아서이고, 그게 분할의 목적입니다.
2. **볼 범위 고르기:** 지도를 보고 "어느 결론·쟁점을 볼지" 하나를 고릅니다.
3. **심층 단계:** **S0는 그대로 두고**(끄지 마세요) [S2 선별](prompts/LARP_split_S2_select.md)과 [기준·점검 모듈](prompts/LARP_modules.md)을 *추가*로 넣습니다. → "[고른 범위]의 이상 논증 검토"라고 지시하면 → 그 부분만 깊이 봅니다.
4. 1단계에서 나온 지도·게이트0 결과는 지우지 말고 두면 다음 단계로 *가져갈 자료*가 됩니다.

(통째로 다 넣어도 되는 넉넉한 환경이라면 분할판 대신 그냥 [`LARP.md`](prompts/LARP.md) 전체판을 쓰세요 — 분할판은 *어쩔 수 없이 쪼개야 하는* 환경용입니다.)

### 5.1 분석 중 외부 확인이 필요할 때 — 딥리서치 물음을 받아내는 법
글 하나로는 답이 안 나오는 대목(예: 인용된 판례가 진짜인지, 인용이 원문과 같은지, V로 표시된 빠진 증거)에서 도구는 단정하지 않고 **확인 물음을 스스로 만들어** 줍니다. 받아내는 법:

1. **그냥 분석을 돌린다.** 확인이 필요한 곳이 나오면 도구가 그 자리에 물음을 붙입니다 — 따로 시키지 않아도 됩니다. (안 나오면 "확인이 필요한 것들을 딥리서치 물음으로 만들어줘"라고 한 줄 요청.)
2. **물음은 두 종류로 나옵니다.** *공개자료형*(법·판례·통계 등)은 **그대로 복사해 딥리서치 AI에 붙일 완결된 질문**으로, *사건기록형*(원문 대조·계좌·시간표 등)은 딥리서치로 못 구하니 **"내 자료의 이 부분을 확인하라"**로 나옵니다.
3. **답을 출처와 함께 다시 붙인다.** 그러면 도구가 반영합니다(출처 없는 답은 사실로 받지 않습니다).

### 5.2 주장이 무엇인지부터 모아야 할 때 — 획득용 딥리서치 물음을 받아내는 법
뉴스·SNS 주장처럼 *원문 없이 흩어진 주장*은, 근거를 직접 찾기 전에 **LARP에게 모으는 물음부터 만들게** 할 수 있습니다. 받아내는 법:

1. **원문 대신 주장만 넣고 "원문이 없으니 근거부터 모아줘"라고 한다.** 그러면 도구가 그 주장의 *가장 강한 형태*와 *권위 있는 반박*을 함께 모으는 딥리서치 물음을 만들어 줍니다(허수아비·저질 출처 방지 장치 포함).
2. **그 물음을 딥리서치 AI에 붙여 자료를 회수한다.** 회수한 자료가 '분석 대상'이 되어 정상 분석으로 들어갑니다.
3. **이후는 §1대로.** 숨은 전제·빠진 증거를 보고, 도구가 다시 만들어 준 확인 물음(§5.1)을 딥리서치로 돌린 뒤, 채택은 내가 정합니다.

딥리서치→LARP, LARP→딥리서치 어느 순서로 돌아도 됩니다.
→ 예시: ["백신은 효과가 없다" 주장 검증](examples/claim_check_vaccine.md)

### 5.3 빠뜨린 증거·지어낸 인용을 코드로 한 번 더 잡기 — 검증 층 (선택)
분석 자체가 이미 *빠진 증거·누락 후보*를 내놓으니 평소엔 그걸로 충분합니다. 다만 **빠지면 큰일 나는 일**이면, 모델 혼자 못 잡는 두 위험(*분석이 놓친 빠짐*, *원문인 척 지어낸 인용*)을 바깥에서 한 번 더 봅니다.

1차 출력을 텍스트 파일로 저장한 뒤:

```bash
# ① 인용이 원문에 실제로 있나 (지어낸 인용 잡기)
python tools/larp_quote_audit.py --source 원문.txt --analysis 1차출력.md
# ② 인용된 증거가 다 들어왔나 (빠짐 잡기)
python tools/larp_coverage_audit.py 원문.txt --tree 1차출력.md
python tools/larp_card_audit.py 1차출력.md
```

그리고 **빠짐 사냥(누락 사냥) 한 번 더** — *새 창이나 다른 모델*에 [`prompts/LARP_verify.md`](prompts/LARP_verify.md)+원문+1차 출력을 넣으면 '아직 안 올라온' 약점·증거·반론만 받아냅니다(같은 대화에서 시키면 자기가 놓친 것을 스스로 못 봅니다). 자세한 빠짐 검사 옵션은 [tools/coverage_audit.md](tools/coverage_audit.md). **파이썬이 부담되면** 코드 실행되는 챗봇(ChatGPT·Claude)에 스크립트와 파일을 올려 돌리면 설치 없이 됩니다.

검증 층은 위험을 *없애지* 않고 *보이게* 만들 뿐 — 마지막 판단은 사람입니다.

### 5.4 어느 모델로, 어디까지 돌릴까 — 실제 측정 결과 (2026-07, 판결문 57쪽 범위)

실제 판결문 하나에 정답지를 만들어 놓고, 모델과 절차의 조합마다 "빠짐없이 건졌는가"(재현율)를 재 본 결과입니다(상세: `verification/cases/case4_2024no620_loop/`).

| 조합 | 재현율 |
|---|---|
| 소형(하이쿠급) 모델 + 프롬프트만 | 56% |
| 소형 + 코드가 뽑은 증거 번호표 목록을 입력(게이트0) | 86% |
| 소형 + 코드로 빠짐 검사 → 빠진 것 되먹여 고침(보수 패스) | 89% |
| 위 + 다른 모델의 빠짐 사냥 | 95.6% |
| 중형(소넷급) 모델 + 프롬프트만 | 96% |

여기서 나오는 실용 지침:

- **모델 등급**: 지도 그리기와 빠짐 사냥은 중형(소넷급)이 실제로 쓸 만한 최소선. 미심쩍은 논증 선별과 정밀 독해는 상위(오퍼스급) 권장. 소형은 빠진 것 고치기 같은 보조 역할에만.
- **모델이 좋아도 검사를 건너뛸 수는 없습니다.** 어떤 등급이든 모델은 자기가 빠뜨린 것을 말해 주지 않고, 스스로 센 "목록 다 처리했다"는 보고도 믿을 수 없습니다(실제 측정: 52/54 처리했다고 보고했지만 진짜로는 26/54). 세는 일은 코드가 해야 합니다.
- **어디까지 돌릴지는 "얼마나 믿고 쓸 것인가"로 정합니다**: 훑어보기 → 코드 대조 + 보수 패스까지(비용 거의 없음, 사실상 기본). 판단·문서에 쓸 결과 → 별도 모델 누락 사냥까지. 생략한 층이 있으면 출력은 '미검증'으로 남습니다(게이트4).
- 전부 다 돌려도 빠짐은 조금 남습니다(측정 약 4% — 번호표 없는 증거, 뜻을 읽어야만 잡히는 빠짐). 그 마지막 몫은 사람 검토입니다.

## 6. 자주 묻는 것

**Q. 도구가 "이 글은 틀렸다"고 말해 주나요?** 아니요. "여기가 의심스러우니 살펴보라"고 *자리만* 짚습니다. 받아들일지는 사람이 정합니다.

**Q. 지적이 너무 많거나 적어요.** "좀 더 보수적으로 / 좀 더 공격적으로 골라줘"라고 하면 조절됩니다. 제외한 것의 이유도 함께 보여 주니 그 목록도 보세요.

**Q. 형사 사건 문서가 아니어도 되나요?** 됩니다. 기사·보고서·투자 판단, 내 생각에도 똑같이 씁니다. 형사 전용은 '합리적 의심 보고서' 하나뿐.

**Q. 결과가 너무 어려워요.** 맨 앞 *쉬운 요약*을 먼저 보세요. 그래도 어려우면 "중학생도 이해하게 쉽게 풀어서 설명해줘"를 덧붙이면 됩니다.

**Q. 1차에서 안 멈추고 끝까지 가버려요.** 첫 메시지에 **"1차(논증 지도)까지만 하고 멈춰"**를 덧붙이세요. 작은 모델일수록 이 한 줄이 도움이 됩니다.

**Q. 보안은?** **실제 사건기록·송치자료는 외부 AI에 넣지 마세요.** 연습은 공개된 글로.
